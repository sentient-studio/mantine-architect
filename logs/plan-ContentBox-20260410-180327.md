## 1. 🚩 Architectural Conflicts & Recommendations

### A — Component Cannibalization
**Severity: None detected**
ContentBox uses Mantine `Box` (for static default/swap) and `UnstyledButton` (for link) — both are the correct primitives. No Mantine widget is being duplicated.

### B — Layout Paradox
**Severity: 🟡 ADAPT**
- **Design Problem:** Figma specifies fixed width — 635px in the component definition, 440px on the canvas instance. Both are hardcoded, coordinate-based widths.
- **Framework Solution:** `width: 100%` — ContentBox is a responsive container that fills its parent. A fixed 440px would break in narrower layouts and is not appropriate for a general-purpose container primitive.
- **Trade-off:** The component is visually wider/narrower depending on context, which is correct for production. The Figma artboard width is a preview aid, not a design constraint.

### C — Accessibility Tension
**Severity: None detected**
Text fill is #000000 (black) on gray.0 (#f8f9fa) background. Contrast ratio ≈ 17:1 — well above WCAG AA (4.5:1). `variant=Link` renders as `<button>` (not `<div onClick>`), providing keyboard operability by default. No overrides needed.

### D — Thin Wrapper Docs Gap
**Severity: 🔵 NOTE**
`ContentBoxProps extends React.HTMLAttributes<HTMLElement>`. Storybook can introspect native interfaces, but all ~50 HTML attributes will flood the Controls panel. Plan for explicit `argTypes` to surface only the meaningful props (`variant`, `children`) and suppress the noise.

### E — Design Omissions & Visual Deviations
**Severity: 🟡 ADAPT** (3 items)

**E-1: Background fill omitted in previous implementation**
- Design Problem: All three Figma variants carry a permanent solid gray.0 fill (`#f8f9fa`, variable `VariableID:...8641:729`). The fill is always visible, not conditional.
- Implementation: Previous code had `background: transparent`. The component rendered as a transparent surface with only hover states showing any background colour — the opposite of what Figma shows.
- Rationale: Bug in the previous generation. This regeneration corrects it to `background-color: light-dark(var(--mantine-color-gray-0), var(--mantine-color-dark-7))` as the persistent base.

**E-2: Gap omitted in previous implementation**
- Design Problem: All three Figma variants set `itemSpacing: 12px` — the gap between flex children on the primary (horizontal) axis.
- Implementation: Previous code had `gap: 0`. Multi-child layouts (icon + label, avatar + text) were collapsed together.
- Rationale: Bug in the previous generation. Corrected to `gap: var(--mantine-spacing-sm)` (12px).

**E-3: `variant=Swap` not exposed as a prop value**
- Design Problem: Figma Component Set contains a distinct `variant=Swap` with a `_Swap Placeholder` child node (107:1417). Designers may expect this to map to a named API option.
- Implementation: `_Swap Placeholder` is a Figma component-swap design-tool artifact — it signals that the slot accepts any component, not a literal production concept. In React, `children: ReactNode` already covers all composition needs. Exposing a `'swap'` variant string would add API surface with no behavioural difference from `'default'`.
- Rationale: Correct omission. The `WithRichContent` story demonstrates this compositional use case.
