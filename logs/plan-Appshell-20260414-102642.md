## 1. 🚩 Architectural Conflicts & Recommendations

### A — Component Cannibalization
**Severity: None detected**  
The design maps directly to Mantine's AppShell compound API — `AppShell`, `AppShell.Header`, `AppShell.Navbar`, `AppShell.Main`, `AppShell.Aside`, `AppShell.Footer`. We are wrapping Mantine's own component; no custom layout implementation is required.

### B — Layout Paradox
**Severity: None detected**  
All zone dimensions in Figma (header 60px, footer 60px, navbar 300px, aside 300px) translate directly to Mantine's AppShell configuration props (`header={{ height: 60 }}`, etc.). No absolute-to-flow conversion is needed.

### C — Accessibility Tension
**Severity: 🔵 NOTE**  
The Figma uses grape-colored slot indicators (`--grape/grape-3`: #e599f7, ~1.97:1 on white) to mark the five content zones. These are Figma design-tool artifacts with no production meaning — the shell itself contains no rendered text, so no contrast override is needed. AppShell's compound components render the correct HTML5 landmark elements natively (`<header>`, `<nav>`, `<aside>`, `<footer>`, `<main>`). The Burger toggle will carry `aria-label="Toggle navigation"`.

### D — Thin Wrapper Docs Gap
**Severity: 🔵 NOTE**  
The component uses a bespoke explicit prop interface (not `extends AppShellProps`), so Storybook can introspect the types directly. Explicit `argTypes` are still required for config props (`withAside`, `headerHeight`, `padding`, etc.) to produce a useful Controls panel on the docs page. A `Showcase` story using `args:` is mandatory.

### E — Design Omissions & Visual Deviations
**Severity: 🟡 ADAPT**

**Entry 1 — Aside height**
- Design Problem: Figma renders the Aside at exactly 780px (900px canvas − 60px header − 60px footer). This is a static artboard arithmetic value.
- Implementation: Mantine AppShell calculates Aside height automatically as `100vh − header − footer`. A hardcoded 780px would break at any other viewport height.
- Rationale: The designer's intent is clearly "fill the available height between header and footer" — the 780px is a design-tool artifact, not a production specification.

**Entry 2 — Mobile responsive behavior**
- Design Problem: Figma provides only a 1440px desktop layout. No mobile variant or Burger toggle is shown.
- Implementation: A Burger toggle and mobile Navbar collapse (`collapsed: { mobile: !opened }`) will be added, managed via internal `useDisclosure`. This is standard for any production AppShell.
- Rationale: A layout shell without mobile collapse behavior is not production-grade. The desktop Figma implies a responsive shell structure.

## 8. Stories List

All AppShell stories require `parameters: { layout: 'fullscreen' }` — the shell fills the iframe viewport via `position: fixed`.

1. **Showcase** *(primary docs story, uses `args:`)* — Full 5-zone shell with all config props wired to Storybook Controls. Default args: `withAside: true`, `withNavbar: true`, `headerHeight: 60`, `footerHeight: 60`, `navbarWidth: 300`, `asideWidth: 300`, `padding: 'md'`. Representative slot content: nav link list in Navbar, data-table placeholder in Main, widget stack in Aside, status bar in Footer.

2. **WithoutAside** — `withAside={false}`: 4-zone shell (header + navbar + main + footer). Demonstrates the boolean prop; verifies layout recalculates without Aside offset.

3. **WithoutNavbar** — `withNavbar={false}`: 4-zone shell (header + main + aside + footer). No Burger shown in header since navbar is disabled.

4. **Minimal** — `withNavbar={false} withAside={false}`: 3-zone shell (header + main + footer only). Verifies minimum viable shell configuration.

5. **Open** *(Playwright fixture — `parameters: { docs: { disable: true } }`)* — Full shell in desktop layout state for axe and visual snapshot tests. Avoids Burger interaction in CI.

## 10. Ambiguities Resolved

| Decision | Rationale |
|---|---|
| **`withAside` boolean prop (default `true`)** | Sibling Figma node `133:1082` implies an "Aside No" state; boolean prop covers both layouts without needing two components |
| **Mobile Burger toggle added** | Figma desktop-only design implies a responsive shell; production AppShell requires mobile collapse behavior — using Mantine's `useDisclosure` + `Burger.hiddenFrom="sm"` |
| **Aside 780px → responsive height** | Design-frame arithmetic (900px canvas − 60px header − 60px footer); Mantine calculates this automatically; hardcoding 780px would be a regression |
| **Grape slot indicators omitted** | `--grape/grape-3` (#e599f7) values are Figma design-tool markers with no production mapping; confirmed by Figma's use of `--spacing/spacing-md` variable names in the same layer |
| **Slot content as `ReactNode` props** | Simpler than compound sub-component exports; Storybook Controls work cleanly; consumer composes their own content using Mantine primitives |
| **Internal `useDisclosure` for mobile state** | Shell manages its own toggle lifecycle; no need to expose `opened`/`onToggle` for the wrapper's primary use case; stories render at desktop viewport so toggle is not exercised |
| **`withBorder` default kept** | Mantine's `withBorder` default (`true`) produces `1px solid var(--mantine-color-gray-3)` which exactly matches Figma borders — no CSS override required |
| **`position: fixed` kept (Mantine default)** | Expected AppShell behavior; works correctly in Storybook iframe context (iframe IS the viewport) |
| **No Storybook decorator for height** | `parameters: { layout: 'fullscreen' }` + AppShell's `position: fixed` fills the iframe natively; no `100vh` wrapper div needed |
