## 1. 🚩 Architectural Conflicts & Recommendations

### A — Component Cannibalization
**Severity: None detected**

ContentBox is a generic styled container. Mantine's `Paper` (elevation/shadow surface) and `Card` (structured header/body/footer slots) are the closest built-ins, but neither matches: the design has no shadow and no structural slots. The correct approach is Mantine `Box` (div) as the static root and `UnstyledButton` as the interactive root, both styled via CSS Modules. No custom widget is duplicating a Mantine component.

### B — Layout Paradox
**Severity: None detected**

The Figma component uses flex layout with `align-items: center` and `justify-content: center` — these map directly to CSS flexbox with no absolute-positioning conversion needed. Container width is responsive (width: 100% in code, not the frame width from Figma).

### C — Accessibility Tension
**Severity: 🔵 NOTE**

- The **Link variant** must render as a `<button>` (via `UnstyledButton`), not a `<div onClick>`, to ensure keyboard operability and correct ARIA role. This is an implementation rule, not a Figma conflict.
- Text colour `var(--text/title, black)` on `var(--background/default-hover, #f8f9fa)` ≈ black on white — contrast ratio >10:1 ✅. No override required.
- No colour contrast issues detected for hover state (gray.1 on white background context is not interactive text).

### D — Thin Wrapper Docs Gap
**Severity: None detected**

ContentBox is built from scratch with a concrete prop interface — no `extends MantineXxxProps` pattern. Storybook will introspect the local type definition without issue.

## 5. Props List

```typescript
interface ContentBoxProps {
  /** Content to render inside the box */
  children?: React.ReactNode;
  /** Visual and behavioural variant. Default: 'default' */
  variant?: 'default' | 'link';
  /** Click handler — only meaningful for variant="link" */
  onClick?: React.MouseEventHandler;
  /** Additional CSS class merged with root */
  className?: string;
}
```

- `children` — arbitrary React content; replaces Figma placeholder text and Swap slot.
- `variant` — drives root element switch (`Box` div vs `UnstyledButton`) and CSS `data-variant` selector.
- `onClick` — only attached to `UnstyledButton` root when `variant="link"`.
- `className` — merged via `cx()` for Styles API / consumer overrides.
- Standard HTML props (`id`, `style`, `aria-*`) passed through via `...rest` spread to the root element.

## 10. Ambiguities Resolved

| Question | Decision | Rationale |
|---|---|---|
| Should Figma's "Swap" variant become a code variant? | **No.** Omit from implementation. | The `_Swap Placeholder` layer has an underscore prefix (Figma convention for non-deliverable utility layers) and renders a grape-coloured `"Slot"` indicator. In code, the `children` prop is the composition mechanism — no separate variant needed. |
| Should the component have a fixed width? | **No — width: 100%.** | The Figma frame shows absolute widths (440–635px) that are frame-measurement artefacts, not design intent for the component. A content container should fill its parent. |
| Should the Link variant accept `href` for anchor navigation? | **Via polymorphic spread only.** | The component will accept `...rest` which Mantine's `UnstyledButton` will forward including `component="a" href="..."`. No first-class `href` prop is needed; it adds surface area without clear benefit for this thin a wrapper. |
| Should dark mode backgrounds be included? | **Yes.** | The container is a surface element — dark mode tokens ensure it adapts correctly in themed apps. Adds negligible complexity and avoids silent failures. |
| Should the component export a display name for Storybook? | **Yes** — `ContentBox.displayName = 'ContentBox'` | Ensures correct autodocs labelling since the Figma node is named "Box". |
