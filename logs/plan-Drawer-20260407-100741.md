| Decision | Rationale |
|---|---|
| Thin wrapper (not custom overlay) | Mantine's Drawer covers ‚â•80% of the design including focus trap, scroll lock, portal, overlay, keyboard handling. Conflict Category A applies. |
| `position="right"` as default | Figma canvas positions the panel at x=520 (right side). Mantine's default of `"left"` doesn't match the design. |
| `size=400` as default | Figma panel is exactly 400px wide. Passed to `--drawer-size` internally. |
| `shadow="lg"` as default | Figma's 3-layer shadow (`0 7px 7px rgba(0,0,0,.04)`, `0 10px 15px rgba(0,0,0,.1)`, `0 1px 3px rgba(0,0,0,.05)`) is deeper than Mantine's `md`. Custom rgba not used in CSS due to token compliance rules ‚Äî closest Mantine shadow token is `lg`. |
| No custom CSS shadow | Token compliance gate forbids raw rgba(). Mantine's `shadow="lg"` is the correct approach via prop, not CSS. |
| Portal CSS scope | No custom CSS variables defined on `.root`. All CSS classes (`content`, `header`, etc.) are portal-rendered Mantine slots. Variables are defined directly within each class rule to avoid portal inheritance failure. |
| `classNames` merging (fix) | Current code uses `Omit<MantineDrawerProps, 'classNames'>` which breaks the Styles API for consumers. Stage 2 must restore `classNames` to the public type and merge consumer overrides after component defaults. |
| Quality gate #6 (data-*) | The Drawer is a thin wrapper delegating all state to Mantine. Mantine internally uses `data-*` on its rendered elements (e.g. `data-opened`, `data-with-close-button`). The component itself does not need custom `data-*` attributes since it has no custom stateful CSS variants. Gate #6 verifies the pattern; thin wrappers satisfy it via Mantine's own data attributes. |
| Story content uses Mantine-native inputs | The Drawer's shell is the component under test. Form content inside stories is illustrative. Using Mantine's `TextInput`, `PasswordInput`, `Checkbox` directly (not re-using project custom components) keeps the Drawer story self-contained. |
| `PasswordInput` left icon | Figma shows lock icon (left) + eye icon (right) on password fields. Mantine's `PasswordInput` natively provides the eye toggle. Adding `leftSection={<IconLock size={14} />}` gives the lock on the left without fighting Mantine's internals. |

## 12. Architectural Conflicts & Recommendations

### A ‚Äî Component Cannibalization
**Severity: üî¥ BLOCK (resolved ‚Äî using Mantine Drawer)**

- **Design Problem:** The Figma shows a slide-in panel with overlay, focus trapping, keyboard dismiss, and sticky header ‚Äî a complete custom overlay widget.
- **Framework Solution:** Mantine's `Drawer` covers this 100%. Use it as the base component, apply overrides via `classNames` prop and CSS module targeting Mantine's named slots.
- **Trade-off:** Mantine's Drawer gives free portal rendering, scroll lock, focus trap, Escape dismiss, `aria-labelledby`/`aria-describedby` wiring, and animation. A custom overlay would require all of this from scratch with high a11y risk.
- **Resolution:** Thin wrapper pattern adopted. No custom overlay implementation. ‚úÖ RESOLVED ‚Äî no human approval needed; Mantine component is in use.

### B ‚Äî Layout Paradox
**Severity: üü° ADAPT**

- **Design Problem:** Figma node has fixed dimensions (400√ó1024px) at absolute canvas coordinates (x=520, y=23). Form fields use explicit pixel gaps (`padding-xs: 12px`, `spacing-sm: 8px`, `spacing-xl: 24px`).
- **Framework Solution:**
  - Drawer width ‚Üí `size={400}` prop (maps to `--drawer-size: 400px`)
  - Drawer height ‚Üí Mantine stretches to viewport height automatically (correct web behavior)
  - Form layout ‚Üí `Stack gap="md"` + `Group grow` for side-by-side fields
  - Gaps ‚Üí `var(--mantine-spacing-*)` tokens, not hardcoded px
- **Trade-off:** Web flow layout is responsive and accessible; fixed Figma coordinates are a canvas artifact. `size={400}` preserves exact 400px width.
- **Pixel delta:** Height 1024px (Figma) ‚Üí 100vh (Mantine stretch). Visual form content ends at ~420px; remaining space is Mantine's scrollable body ‚Äî correct behavior.

### C ‚Äî Accessibility Tension
**Severity: üîµ NOTE**

1. **"Have an account? Login" text ‚Äî gray.6 fails contrast**
   - Figma: `#868e96` (gray.6) = 3.15:1 on white ‚ùå
   - Override: `c="gray.7"` (`#495057`) = 7.45:1 ‚úÖ
   - Applied in `RegistrationForm` story helper.

2. **Required asterisk color ‚Äî red.6 fails contrast**
   - Figma: `text/error/default` = `#fa5252` (red.6) = ~3.5:1 ‚ùå
   - Source: Mantine's `--mantine-color-error` token (internal to TextInput/PasswordInput)
   - Decision: Accept Mantine's default. The asterisk is a supplementary indicator (the label text itself conveys the field identity). Overriding per-input would require the Styles API on every input instance inside the story ‚Äî disproportionate for a decorator character. WCAG SC 1.4.3 compliance is borderline; flag as acceptable risk for a required indicator.

3. **Close button tap target**
   - Figma: 24√ó24px close button
   - Mantine's ActionIcon default minimum is 28px; Mantine Drawer's close button uses a reasonable target size. No explicit override needed.

### D ‚Äî Thin Wrapper Docs Gap
**Severity: üîµ NOTE**

- **Design Problem:** `DrawerProps = MantineDrawerProps`. Storybook autodocs cannot introspect `MantineDrawerProps` from the external `@mantine/core` package ‚Äî the controls panel will show no props.
- **Framework Solution:** Add explicit `argTypes` in story meta for all meaningful props: `position`, `size`, `withOverlay`, `withCloseButton`, `title`. Hide `opened` and `onClose` with `table: { disable: true }`. Include a `Showcase` story with `args:` for Controls to work.
- **Status:** Already addressed in current implementation. Verify in Stage 3 that the Controls panel is populated. ‚úÖ
